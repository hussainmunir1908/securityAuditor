3. SUPABASE DATABASE SCHEMA
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  public.profiles           ← GitHub user data, login, avatar_url, github_access_token, github_id
  public.repositories       ← repo_url, repo_name, user_id (FK profiles), ingestion_status enum
                               ingestion_status values: 'pending' | 'processing' | 'completed' | 'failed'
  public.scan_results       ← repository_id (FK), rule_id, severity, description, remediation,
                               file_path, start_line, end_line, code_snippet
  public.code_chunks        ← repository_id (FK), file_path, start_line, end_line, content,
                               language, chunk_index, embedding vector(1536)
  public.security_rules     ← rule_id (unique), title, content, severity, owasp_category, cwe_id,
                               embedding vector(1536)

  All tables: RLS ENABLED. Users can only read/write their own data.
  Vector indexes: HNSW with vector_cosine_ops on code_chunks.embedding and security_rules.embedding.

  AUTH (no auth required):
    GET  /api/auth/github              → Redirects user to GitHub OAuth consent page
    GET  /api/auth/github/callback     → Handles GitHub OAuth callback, issues JWT cookie
    GET  /api/auth/me                  → Returns current user profile from JWT
    POST /api/auth/logout              → Clears the access_token cookie

  INGESTION (requireAuth middleware):
    POST /api/ingest/github            → Body: { repositoryId }. Ingests via Octokit. Returns 202.
    GET  /api/ingest/status/:repoId    → Returns ingestion_status for a repository
    GET  /api/ingest/repositories      → Lists all repositories for the current user

  UPLOAD (requireAuth middleware):
    POST /api/ingest/upload            → multipart/form-data: file (.zip) + repositoryName. Returns 202.

  SCAN (requireAuth middleware — all STUB):
    POST /api/scan/start               → [STUB] Will trigger SAST scan on a completed repository
    GET  /api/scan/results/:scanId     → [STUB] Will return vulnerability report
    GET  /api/scan/history             → [STUB] Will list all past scans
    GET  /api/scan/metrics/:scanId     → [STUB] Will return aggregate metrics

8. WHAT AWAITS — STEP 3: THE SAST SCANNING ENGINE
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  This is the core intelligence of the platform. Currently all /api/scan/* endpoints
  return 501 Not Implemented. Step 3 will flesh these out fully.

  WHAT NEEDS TO BE BUILT:

  A. RAG Retrieval Pipeline (backend/src/utils/rag.ts — to be created)
     - For each code chunk in a repository, use pgvector cosine similarity search
       to find the most relevant security_rules from the knowledge base.
     - SQL: SELECT * FROM security_rules ORDER BY embedding <=> $1 LIMIT 5
     - This links vulnerable code patterns to their OWASP/CWE definitions.

  B. LLM Analysis Engine (backend/src/utils/analyzer.ts — to be created)
     - Feed the code chunk + retrieved security rules into an LLM (GPT-4o or similar).
     - Craft a structured prompt that instructs the model to identify:
         → Vulnerability type (maps to rule_id / CWE)
         → Severity: critical / high / medium / low
         → Exact line range
         → Description of the vulnerability
         → Concrete remediation advice
     - Parse the LLM response into a ScanFinding object.

  C. scan.ts Route Implementation (backend/src/routes/scan.ts — to be overhauled)
     - POST /api/scan/start:
         1. Validate repositoryId + check ingestion_status === 'completed'
         2. Fetch all code_chunks for the repository
         3. For each chunk → RAG retrieval → LLM analysis → collect findings
         4. Insert findings into scan_results table
         5. Return scanId
     - GET /api/scan/results/:scanId — fetch scan_results by scan_id
     - GET /api/scan/history         — list all scans for user
     - GET /api/scan/metrics/:scanId — count findings grouped by severity + owasp_category

  D. Optional: Real-time Progress via SSE (Server-Sent Events)
     - Stream scan progress to the frontend as each file is analyzed.

  Start Backend (Terminal 1):
    cd backend
    npm run dev
    → Runs on http://localhost:5000

  Start Frontend (Terminal 2):
    cd frontend
    npm run dev
    → Runs on http://localhost:3000

  Seed Knowledge Base (after adding OPENAI_API_KEY):
    cd backend
    npm run seed:rules

  TypeScript Check:
    cd backend && npx tsc --noEmit
    cd frontend && npx tsc --noEmit

  Push to GitHub (only when user explicitly asks):
    git add .
    git commit -m "..."
    git push origin master

12. KEY ARCHITECTURAL DECISIONS & CONSTRAINTS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  - Auth: Backend-managed JWT. Cookie name: 'access_token'. HttpOnly, SameSite=Strict.
    The frontend reads auth state by calling GET /api/auth/me.
    The proxy.ts only checks for cookie *presence* — the actual JWT is verified by the backend.

  - Supabase: Use anon key only. All data access is mediated by RLS policies.
    NEVER use the service_role key in this app.

  - Embeddings: The 'embedding' column in code_chunks and security_rules is vector(1536).
    This matches text-embedding-3-small's output dimension exactly.

  - Ingestion is async: Both GitHub and upload routes return HTTP 202 immediately.
    The actual chunking/embedding work happens in a background IIFE after the response is sent.
    Poll /api/ingest/status/:repoId to check completion.

  - Chunker does NOT use an AST parser (no Tree-sitter). It uses brace-depth tracking.
    This is fast and language-agnostic but may occasionally mis-detect block boundaries
    in files with complex nesting (e.g., deeply nested ternaries, template literals).

  - CSS / Next.js 16 quirks:
    * Use proxy.ts (NOT middleware.ts). Export function must be named `proxy`.
    * No @import url() after @import "tailwindcss" in globals.css — Tailwind v4
      expands the import into real CSS rules, making subsequent @import invalid.
    * Fonts must be loaded via next/font, not Google Fonts @import.

  - Git: Only push when user explicitly requests it. OPENAI_API_KEY is in .env which is gitignored.
